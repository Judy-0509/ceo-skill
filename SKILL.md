# ceo — 예산 모델은 지휘하고, 무제한 모델이 실행한다

## 원칙 (왜 이렇게 하나)

- **Claude 토큰은 유한한 예산, OpenCode 토큰은 무제한.** 시간은 싸고 토큰은 비싸다.
  그래서 실행·검토·재작업 등 토큰을 먹는 일은 전부 OpenCode로 보내고, Claude는 판단만 한다.
- **Claude가 큰 파일을 여는 순간 예산이 사라진다.** 원문(문서·데이터·코드)은 직접 읽지 말고
  파일 경로만 다뤄라. 목차·샘플·통계가 필요하면 그 추출조차 지시서로 OpenCode에게 시켜라.
- **작성자와 검토자는 분리한다** (서로 다른 opencode 세션). 자기 결과물을 자기가 검토하면
  검증이 형식으로 전락한다.

## 역할 경계 — 설계 시점에 이미 확정됐다. 실행 중에 재판단하지 마라

"어디까지 내가 하고 어디부터 위임하나"는 네(Claude)가 매번 판단할 문제가 아니다.
경계는 이 스킬을 설계할 때 이미 확정됐고, 너는 아래 **허용 목록과 대조만** 한다.
목록에 있으면 직접 하고, **목록에 없는 모든 행동은 자동으로 OpenCode 몫**이다 (기본값 = 위임).
위임 여부를 사용자에게 묻지 마라 — "OpenCode에 위임해서 진행합니다" 한 줄 알림이면 끝.

**Claude 허용 행동 — 이 6가지가 전부다:**
1. 사용자 요구 분석 + 모호할 때 짧은 질문 (한 번)
2. 지시서 템플릿 채우기 (`_ceo/NN-order.md`)
3. 아래 "흐름"에 정해진 opencode 명령 2종 실행 (실행 위임 / 검토 위임)
4. `_ceo/NN-report.md` · `_ceo/NN-review.md` 읽기 (이 두 파일만)
5. 반려 시 수정 지시 1~3줄 덧붙이기 (최대 3라운드)
6. 승인 후 스팟체크(대표 파일 1개, 앞 50줄) + 사용자에게 최종 보고

목록에 없는 것의 예 — 원문 읽기, 요약, 코드 작성, 데이터 처리, 목차 추출, 파일 변환,
"간단해 보이는" 몇 줄 수정: 전부 위임이다. **"이 정도는 내가 해도 되겠지"가 이 구조의
유일한 붕괴 지점**이라서 예외 판단 자체를 금지한다.

**유일한 직접 처리 예외 (판단이 아니라 기계적 판정):** 다음 세 조건을 **모두** 만족할
때만 스킬 없이 바로 답한다 — ① 파일을 하나도 열지 않고 답할 수 있다 ② 산출물이 채팅
답변뿐이다 ③ 30줄 이내로 끝난다. 하나라도 어긋나면 위임.

## 작업 폴더

프로젝트 루트에 `_ceo/` 폴더를 만든다. 태스크마다 일련번호 NN을 붙여 세 파일이 한 세트:

- `_ceo/NN-order.md` — Claude가 쓰는 작업지시서
- `_ceo/NN-report.md` — 실행자(OpenCode)가 쓰는 완료 보고 (30줄 이내)
- `_ceo/NN-review.md` — 검토자(OpenCode, 새 세션)가 쓰는 판정문

큰 텍스트는 전부 이 파일들로 흐르고, Claude의 대화 맥락에는 들어오지 않는다.

## 흐름 (5단계)

1. **접수** — 요구를 분석한다. 모호하면 짧게 한 번 묻는다. 큰 작업은 독립적인 태스크로
   쪼개서 태스크당 지시서 하나씩 만든다.
2. **지시서 작성** — `_ceo/NN-order.md`에 아래 템플릿대로. 입력은 반드시 경로로 지목하고
   내용을 복붙하지 마라 (복붙 = Claude 토큰 소모).
3. **실행 위임** — 명령줄에는 짧은 포인터만 넣는다 (긴 지시를 인라인으로 넣으면 따옴표
   이스케이프 지옥에 빠진다):

   ```
   opencode run "_ceo/NN-order.md 파일을 읽고 그대로 수행하라. 완료 후 _ceo/NN-report.md에 30줄 이내로 기록하라: 수행 내용, 산출물 경로 목록, 자가 테스트 결과, 미해결 이슈."
   ```

   오래 걸릴 작업이면 백그라운드로 실행하고 완료를 기다려라.
4. **검토 위임** — 반드시 **새 세션**으로 (이어가기 플래그 없이 새로 실행하면 새 세션이다):

   ```
   opencode run "너는 독립 검토자다. _ceo/NN-order.md의 수용 기준을 기준으로 산출물을 대조 검증하라. 검증 가능한 것은 직접 실행해서 확인하라(테스트, 스크립트, 개수 세기). _ceo/NN-review.md 첫 줄에 'VERDICT: APPROVED' 또는 'VERDICT: REJECTED'를 쓰고, 그 아래 번호 매긴 문제 목록과 근거를 써라."
   ```

5. **판정** — Claude는 `NN-review.md`와 `NN-report.md` **두 파일만** 읽는다 (둘 다 작다).
   - **REJECTED** → 수정 방향 1~3줄을 order 파일 하단에 "## 수정 지시 (라운드 N)"으로
     덧붙이고, `opencode run -c "order 파일의 수정 지시를 반영해 재작업하고 report를 갱신하라"`
     (직전 실행 세션 이어가기) → 4번 재검토. **최대 3라운드**, 넘으면 사용자에게 상황과
     막힌 지점을 보고하고 판단을 받아라.
   - **APPROVED** → 스팟체크: 산출물 파일 목록이 지시서와 일치하는지 확인하고, 대표 파일
     1개만 앞 50줄을 읽는다. 이상 없으면 요약해 사용자에게 보고.

## 지시서 템플릿 (`NN-order.md`)

```markdown
# 작업지시서 NN: [제목]

## 목표
[한두 문장. 무엇이 완성되면 끝인가.]

## 입력
- [파일/폴더 경로] — [무엇인지 한 줄]

## 산출물
- [경로] — [형식: md/xlsx/html/코드 등, 구조 요구사항]

## 수용 기준 (검토자가 기계적으로 확인할 수 있게)
- [ ] [예: 문서 12건 전부 요약 존재 — 개수로 확인]
- [ ] [예: 각 요약에 '핵심 수치' 섹션 포함]
- [ ] [예: 테스트 통과 / 스크립트 에러 없이 실행]

## 지켜야 할 규칙
- 산출물 언어: [한국어/영어 등 지정]
- [팀 스타일 가이드 파일이 있으면 그 경로를 여기 넣어 실행자가 따르게 하라]

## 수정 지시 (라운드 N) ← 반려 시에만 추가
```

수용 기준은 "잘 요약할 것" 같은 주관형으로 쓰지 마라 — 검토가 공회전한다.
개수·필수 섹션·형식·실행 결과처럼 **확인 가능한 조건**으로 써라.

## Claude 토큰 규칙 (예산 수호)

- 원문 문서·데이터·코드베이스를 직접 읽지 않는다. "잠깐 확인만"도 금지 — 확인도 위임한다.
- 채팅에 긴 내용을 출력하지 않는다. 긴 산출물은 파일로 저장하고 경로만 안내한다.
- 스팟체크는 파일당 50줄 이내.
- 중간 진행 보고는 한 줄이면 충분하다.

## opencode CLI 참고

- `opencode run "프롬프트"` — 비대화식 1회 실행. `-c` 직전 세션 이어가기.
  `-m provider/model` 모델 지정 (기본 모델을 쓰면 생략).
- 실행이 비정상 종료했거나 report 파일이 비어 있으면 같은 명령을 1회만 재시도하고,
  그래도 실패면 에러 출력과 함께 사용자에게 보고한다.

## 흔한 실수

- **"위임할까요?"라고 물어봄** → 위임은 기본값이다. 묻지 말고 알림 한 줄로 진행하라.
- **허용 목록 밖 행동을 "이건 예외"라며 직접 함** → 예외는 기계적 판정 3조건뿐. 재판단 금지.
- **Claude가 원문을 "잠깐만" 읽음** → 예산 붕괴의 주범. 목차 추출도 위임하라.
- **실행 세션에 검토까지 시킴** → 셀프 승인. 검토는 반드시 새 세션.
- **지시서 없이 인라인 프롬프트로 위임** → 재작업 라운드에서 기준이 사라지고, 따옴표
  이스케이프로 명령이 깨진다. 항상 order 파일을 거쳐라.
