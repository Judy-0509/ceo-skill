# 표준 레시피 — 지시서에 붙여 넣는 검증된 명령 블록

지시서를 백지에서 설계하지 마라. 아래에서 유형을 골라 **[대괄호]만 채워** order 파일의
해당 칸에 붙여라. 맞는 유형이 없으면 맨 아래 R6 범용 레시피를 쓴다.
(왜: 지시서 품질이 이 구조 전체의 품질을 결정하는데, 그 설계를 매번 즉석에서 하면
저노력 모델이 수용 기준을 빠뜨린다. 레시피는 이미 검증된 설계다.)

## R1. 문서 반입 + 다이제스트 (PDF·DOCX·PPTX, 1건이든 여러 건이든)

가장 흔한 유형. 핵심: **요약을 받는 게 아니라, 이후 질의응답의 근거가 될
다이제스트를 만든다.** Claude은 원문 대신 이 다이제스트만 읽는다.

```
## 목표
[폴더/파일 목록]의 문서를 전부 읽고 다이제스트를 만든다.

## 산출물
- _ceo/NN-digest.md — 문서마다 15줄 이내로: ① 한 줄 정체(무슨 문서·날짜·출처)
  ② 핵심 주장 3~5개 ③ 핵심 수치(값+단위+출처 페이지) ④ 키워드 10개
- (개별 상세 요약이 최종 산출물이면) output/요약-<파일명>.md

## 수용 기준
- [ ] 입력 문서 수 = digest 항목 수 (누락 0 — 개수로 확인)
- [ ] 모든 수치에 출처(파일명+페이지) 표기
- [ ] 읽기 실패한 파일은 건너뛰지 말고 report에 파일명+사유 기록
```

## R2. 엑셀 읽기·취합·계산

```
## 목표
[엑셀 파일들]을 읽고 [취합/집계/비교] 결과를 만든다.

## 산출물
- _ceo/NN-digest.md — 파일·시트 구조: 시트명, 행수, 열 헤더, 표 시작 위치
- output/[결과 파일명].md 또는 .csv — [요구 집계]

## 수용 기준
- [ ] 합계·소계를 재검산해 원본과 일치 (검토자가 직접 재계산)
- [ ] 행 수 보존 확인 (원본 합 = 결과 합, 제외분은 사유와 개수 명시)
- [ ] 빈 셀·오류 셀(#REF! 등) 개수 보고

## 지켜야 할 규칙
- 수식이 아니라 계산된 값 기준으로 읽고, 반올림 규칙을 결과에 명시
- 엑셀 도구는 환경 규칙을 따르라 (팀 가이드가 있으면 그 도구 — 예: xlwings)
```

## R3. 후속 질의응답 — 다이제스트 이후의 탐색 (원문 열람 대체)

Claude이 digest를 읽고도 더 필요한 게 있으면 **절대 원문을 열지 말고** 이 레시피로 묻는다.
`_ceo/NN-qa.md`에 질문을 번호 매겨 추가하고:

```
opencode run -c "_ceo/NN-qa.md의 미답변 질문에 답하라. 각 답은 30줄 이내, 원문 인용 1~2개와 출처(파일명+페이지) 필수. 답은 같은 파일의 해당 질문 바로 아래에 써라. 원문에 없으면 '없음'이라고 써라 — 추정 금지."
```

- `-c`로 **실행 세션을 이어가는 것이 핵심**: 실행자가 이미 읽은 문서 맥락을 재사용하므로
  재반입 비용이 0이다. 세션이 만료됐으면 새 세션에 "digest와 원문 경로를 참고해 답하라"로 대체.
- 왜 이 방식인가: Claude이 원문을 읽으면 그 토큰이 컨텍스트에 상주해 이후 모든 턴마다
  재과금된다. 질의 루프 비용은 문서 크기와 무관하게 질문 수에만 비례한다.

## R4. wiki/DB 구축 — 반드시 2단계

```
[1차 지시서] 스키마/구조 제안만 받아라:
## 목표: [소스]로 만들 wiki/DB의 스키마(필드 정의)와 샘플 항목 3건을 제안하라. 30줄 이내.
→ Claude이 확인 (필요시 사용자에게 1회 확인) →
[2차 지시서] 확정 스키마로 전체 구축:
## 수용 기준
- [ ] 항목 수 = 소스 문서/레코드 수와 대조 일치
- [ ] 필수 필드 채움률 100% (빈 필드는 '없음' 명기)
- [ ] 내부 링크/키 깨짐 0건 (검토자가 스크립트로 확인)
```

왜 2단계: 스키마가 틀리면 전량 재작업이다. 싼 확인 1회가 비싼 반려 3회보다 낫다.

## R5. 코드 작업

```
## 산출물
- 변경/생성 파일 목록 + 실행 방법 (report에)
## 수용 기준
- [ ] 테스트/빌드 통과 — 검토자가 직접 실행해 확인 (통과 주장만으로는 불충분)
- [ ] [요구 기능별 확인 항목 — 입력→기대 출력 형태로]
- [ ] TODO/스텁/test.skip 없음
```

## R6. 범용 — 위에 없는 모든 명령

어떤 요청이든 이 4요소로 변환하면 지시서가 된다:
① 목표 한 문장 ② 입력(경로) ③ 산출물(경로+형식) ④ **검증 가능한 수용 기준 3개 이상**.

수용 기준을 못 만들겠으면 최소셋: "산출물에 [필수 섹션]이 존재한다" + "report에
자가 점검 결과를 항목별로 기록한다". 산출물이 클수록 채팅이 아니라 파일로 받고,
Claude이 읽을 부분(digest·report)은 항상 줄 수 제한을 걸어라.
